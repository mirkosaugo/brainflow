How one web architect used Claude Code skills, agents, and orchestration patterns to build an AI-powered brainstorming canvas with Digital Twins — solo.

1. The Vision

A case history in AI-augmented development — not replacing teams, but redefining what one person can orchestrate.

"Da solo so orchestrare tutto quasi come se fossi un team" — but the point isn't that AI replaces people. It's that when human creativity directs AI execution, one person can deliver across design, engineering, documentation, and presentation with a consistency that was previously only possible with a coordinated team.

What is BrainFlow?

BrainFlow is a canvas-based brainstorming tool where you build visual flows of ideas and let AI synthesize them. Create text notes, concept cards, goal cards, perplexity cards, and Digital Twins — AI personas with programmable behaviors (devil's advocate, collaborator, analyst, provocateur). Connect everything to a Run Flow node for AI synthesis, then chain the output into new flows for iterative deepening.

The Product

Infinite canvas with Digital Twins, multiple card types, glassmorphic UI, real-time AI synthesis, chainable flows, and ambient visual effects.

2. The Stack

Modern tooling, intentional choices. Each dependency earned its place.

Framework

Next.js 16.2 (App Router) with React 19 and TypeScript 5 strict mode. Server-side API routes for Claude integration.

Canvas

@xyflow/react 12 (React Flow v12) — node graph system with custom nodes, edges, connection logic, and smart handle positioning.

Styling

Tailwind CSS v4 + shadcn/ui + class-variance-authority. Glassmorphic design system with backdrop-blur, oklch colors, and radius scale.

AI

@anthropic-ai/sdk — Claude API for creative synthesis. Connected nodes are collected and processed through a system prompt tuned for artists.

3. The AI Orchestration Model

Six custom skills encode project standards. Agents execute them. The human architect directs.

Custom Skills

Each skill is a structured prompt in .claude/skills/ that encodes the project's design system, architecture rules, and quality standards. When invoked, the AI agent operates within these constraints — producing consistent results across different sessions and contexts.

/design Any visual task — logos, palettes, graphics — enforcing the exact design tokens (oklch colors, radius scale, glass effects)
/component New UI components — runs a pre-flight checklist (shadcn → existing → base-ui → custom) before writing any code
/node New canvas node types — composes from shared primitives (NodeHandles, NodeHeader, NodeActions, EditableField)
/present Presentation materials — pitch decks, case studies, marketing pages in the Flow brand voice
/audit Systematic review — architecture compliance, design system fidelity, code quality, performance, a11y
/document Decision records — captures why, not just what, in structured ADR format for future reference

Why Skills Matter

Without skills, every AI interaction starts from scratch. You explain the design system again. You remind it about shadcn. You re-specify the node composition pattern. Skills encode this knowledge once and enforce it every time. The result: consistency across days, weeks, and sessions — the kind of consistency that normally requires a team with shared documentation and code review.

4. The Design System

Every color, radius, and shadow is tokenized. The same system powers the app, the docs, and this page.

Color Palette

Purple — #A78BFA

Primary brand accent. Concept cards, creative elements, primary actions.

Cyan — #38BDF8

Media and visual elements. Image nodes, technology references.

Emerald — #34D399

AI and success states. Run nodes, confirmations, growth indicators.

Slate — #64748b

Neutral supporting elements. Text nodes, secondary information.

Glassmorphism

App overlays (header, prompt bar, toolbars, panels) use a glassmorphic treatment: semi-transparent background, backdrop-blur-2xl, subtle border, and a soft shadow. Canvas nodes are exempt — they use solid backgrounds for readability.

/* Glass overlay formula */
background: rgba(255, 255, 255, 0.06);  /* dark mode */
border: 1px solid rgba(255, 255, 255, 0.1);
backdrop-filter: blur(40px);
box-shadow: 0 0 15px 0 rgb(0 0 0 / 0.25);
border-radius: 0.875rem;

Typography

Satoshi for all UI text and headings — a geometric sans-serif with personality. Geist Mono for code and technical content. Both chosen for their clarity at small sizes and their modern, slightly warm character.

6. Lessons & Reusable Patterns

What worked, what to replicate in future projects.

Pattern 1: Encode Standards in Skills, Not Memory

AI memory is useful for preferences, but skills are better for standards. A skill runs the same way every time — it doesn't drift, forget, or interpret differently. The /design skill always checks the same color palette. The /component skill always runs the shadcn pre-flight.

Pattern 2: Composition Over Configuration

Every canvas node is composed from 4 shared primitives. This isn't about DRY for its own sake — it's about making new node types trivial to create while guaranteeing visual and behavioral consistency. The /node skill encodes this pattern so even an AI agent can't shortcut it.

Pattern 3: Glass for Overlays, Solid for Content

The glassmorphism rule is simple but critical: floating UI gets the blur treatment, canvas content stays solid. This preserves readability where it matters (nodes you're editing) while giving the chrome a premium feel. Encoded in the /design and /component skills.

Pattern 4: Document the Why, Not the What

Code shows what changed. Git shows when. Documentation should capture why — the constraints, trade-offs, and alternatives considered. The /document skill produces ADRs (Architecture Decision Records) that capture this context.

Pattern 5: The Human Stays in the Loop

AI orchestration works best when the human makes creative and architectural decisions and the AI handles execution within defined constraints. Skills define the constraints. The architect directs the work. Neither works well alone.

8. How to Replicate This Setup

Step-by-step guide for applying this orchestration model to any project.

  1. Start with CLAUDE.md — Define your tech stack, architecture rules, and design tokens. This is the foundation every AI interaction reads.
  2. Create AGENTS.md — Document the agent roles and orchestration philosophy. This frames how AI should approach work in your project.
  3. Build skills incrementally — Start with /design and /component (the most frequently used). Add more as patterns solidify.
  4. Encode standards, not preferences — Skills should contain hard rules (color tokens, composition patterns). Memory should contain soft preferences (user communication style).
  5. Use memory for feedback loops — When you correct the AI, save it as feedback memory so it doesn't repeat mistakes across sessions.
  6. Document decisions as you go — Use /document to capture why decisions were made. Future-you (and interviewers) will thank you.
  7. Audit periodically — Run /audit to catch drift from the design system and architecture patterns.
